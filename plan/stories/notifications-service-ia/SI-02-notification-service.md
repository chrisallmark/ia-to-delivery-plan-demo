# Implement FCM push provider

## Service

**notification-service** — REST service dispatching transactional notifications across email, SMS, and (soon) push channels. Owned by the Platform team.

## Phase

Phase 2 — Provider & Preferences

## Description

Implement `FirebasePushProvider` in `src/channels/push/` satisfying the `PushProvider` interface from SI-01. The provider uses the Firebase Admin SDK to dispatch push notifications. Firebase credentials must be read from Vault at runtime (environment variable injection — do not hard-code). Credential rotation per environment must be factored into the implementation and documented for Platform Ops. Depends on SI-01 for the `PushProvider` interface.

**Target file:** `src/channels/push/FirebasePushProvider.ts` — a class implementing `PushProvider` from SI-01.

**Firebase Admin SDK:** The SDK is not yet integrated (noted in `CLAUDE.md`). Add `firebase-admin` as a production dependency (`pnpm add firebase-admin`). Initialise the app once at module load using credentials from environment variables (e.g. `FIREBASE_SERVICE_ACCOUNT_JSON` as a base64-encoded JSON string or individual key fields). Follow the same pattern as `SENDGRID_API_KEY` usage in the email channel.

**send() implementation:** Call `admin.messaging().send()` with the device token from `PushPayload` and the message body. Handle FCM error codes — specifically `messaging/registration-token-not-registered` (device token expired; surface as a retriable dispatch error) and `messaging/invalid-argument` (payload malformed; surface as a non-retriable error).

**Factory registration:** Register `FirebasePushProvider` in `src/channels/index.ts` so the factory instantiates it when `'push'` is requested.

**Tests:** Mock the Firebase Admin SDK. Cover: successful dispatch, expired token error, malformed payload error. Do not make live FCM calls in tests.

**Credential prerequisite:** Vault entries for `FIREBASE_SERVICE_ACCOUNT_JSON` (or equivalent) must be provisioned for dev, staging, and prod before integration testing. Coordinate with Platform Ops before starting Phase 2.
