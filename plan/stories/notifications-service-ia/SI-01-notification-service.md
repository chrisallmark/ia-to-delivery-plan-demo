# Add push provider abstraction

## Service

**notification-service** — REST service dispatching transactional notifications across email, SMS, and (soon) push channels. Owned by the Platform team.

## Phase

Phase 1 — Abstraction

## Description

Introduce a `PushProvider` TypeScript interface in `src/channels/push/` following the existing `BaseChannel` pattern. Update the channel factory in `src/channels/index.ts` to resolve `PushProvider` implementations by name at runtime. The interface must be the single abstraction point for all push-specific work; no concrete provider (SI-02) or route (SI-04) may bypass it. This SI does not implement any push dispatch logic — it establishes the contract only.

**New directory:** Create `src/channels/push/` following the structure of `src/channels/email/` and `src/channels/sms/`.

**Interface:** Define `PushProvider` in `src/channels/push/PushProvider.ts` extending or implementing `BaseChannel.ts`. At minimum, the interface should include a `send(payload: PushPayload): Promise<void>` method where `PushPayload` is a new type defined alongside the interface.

**Factory update:** In `src/channels/index.ts`, register `'push'` as a resolvable channel name mapped to the `PushProvider` factory. The factory should throw a clear error if no push provider is configured at runtime (fail fast).

**Tests:** Add a unit test that verifies the factory resolves a stub `PushProvider` implementation correctly when `'push'` is requested.
