# Expose push send endpoint

## Service

**notification-service** — REST service that dispatches transactional notifications across email, SMS, and (soon) push channels. Node.js 20 / TypeScript 5 / Fastify 4 / PostgreSQL 15.

## Phase

Phase 3 — Send endpoint

## Description

Add `POST /notifications/push` endpoint. The handler must: validate the incoming payload; check the recipient's opt-in state from the `push_preferences` table (SI-03) and reject or silently skip if the user has opted out; resolve the push provider via the channel factory (SI-01 / SI-02); dispatch the notification; and return `HTTP 202 Accepted`. Depends on both SI-02 (FCM provider) and SI-03 (preferences table) being merged before this endpoint is testable end-to-end.

- Blocks: SI-05

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Location:** `src/routes/` — add a new route file (e.g. `src/routes/push.ts`) and register it in the Fastify app entry point (`src/index.ts` or the routes index).

**Opt-out enforcement:** Query `push_preferences` before dispatching. If `opted_in = false` or no preference record exists for the user, return `202` without dispatching (silent skip is standard for notification services — confirm this is the intended behaviour with the Platform team). Do not return an error for opted-out users.

**Factory resolution:** Call `src/channels/index.ts` factory with `"push"` to obtain the `FirebasePushProvider`. This keeps the route decoupled from the concrete provider.

**Payload validation:** Use a Fastify schema or a Zod/TypeBox schema for the request body. At minimum validate `user_id`, `title`, `body`, and `device_token` (or derive the token from the preferences table using `user_id`).

**Tests:** Test happy path (202, FCM called), opted-out user (202, FCM not called), invalid payload (400), and FCM provider error propagation.
