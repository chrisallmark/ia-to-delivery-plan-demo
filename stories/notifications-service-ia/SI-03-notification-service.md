# Persist notification preferences

## Service

**notification-service** — REST service that dispatches transactional notifications across email, SMS, and (soon) push channels. Node.js 20 / TypeScript 5 / Fastify 4 / PostgreSQL 15.

## Phase

Phase 2 — Provider & Preferences

## Description

Add a `push_preferences` table to store per-user device tokens and opt-in state. Wire the new table into the existing preferences API so that users can register a device token and set their push opt-in/opt-out preference. The opt-out state stored here is enforced by SI-04's send endpoint — this SI must be deployed before SI-04. Does not depend on SI-02 and can be developed in parallel with it.

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Migration:** Add a new numbered `.sql` file to `migrations/` (e.g. `003_add_push_prefs.sql` — check the existing numbering). Register it in `src/db/migrate.ts`. The table should store at minimum: `user_id`, `device_token`, `opted_in` (boolean), `created_at`, `updated_at`. Consider whether a user can have multiple device tokens (multiple devices) — confirm with the Platform team before designing the schema.

**Preferences API:** The CLAUDE.md describes an existing preferences layer in `src/preferences/`. Extend it with push-specific handlers. Do not duplicate preference logic that already exists for email/SMS — follow the same pattern.

**Route registration:** Add any new API routes (e.g. `POST /preferences/push`, `DELETE /preferences/push/token`) to `src/routes/`. Follow the Fastify route registration pattern.

**Tests:** Test registration of a device token, opt-out (ensures `opted_in = false`), and opt-in reversal. Use Vitest co-located test files.
