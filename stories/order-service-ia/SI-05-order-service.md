# Handle instalment payment webhooks

## Service

**order-service** — REST service managing the full order lifecycle — basket, checkout, payment and fulfilment handoff. Python 3.12 / FastAPI / asyncpg / PostgreSQL 15 / Stripe SDK 9.x.

## Phase

Phase 4 — Webhooks

## Description

Add `POST /webhooks/stripe/instalment` to `payments/router.py`. The handler must: verify the Stripe-Signature header using `stripe.Webhook.construct_event(payload, sig_header, secret)` and return `HTTP 400` on failure; process `payment_intent.succeeded` and `payment_intent.payment_failed` event types; update the corresponding `InstalmentsSchedule` record status (SI-01 repository); and emit a domain event (`instalment.paid` or `instalment.failed`) via `events.emit.publish`. Unrecognised event types should return `HTTP 200` without processing (Stripe best practice). Requires SI-04 to have created instalment schedule records and for `events/schema.py` to be extended with the new event types.

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Location:** `src/payments/router.py` — add route `POST /webhooks/stripe/instalment`.

**Signature verification:** The CLAUDE.md is explicit: use `stripe.Webhook.construct_event(payload, sig_header, secret)` with `STRIPE_WEBHOOK_SECRET` from the environment. Return `400` on `stripe.error.SignatureVerificationError`. Never skip this check.

**Event schema extension:** `events/schema.py` must have `instalment.paid` and `instalment.failed` Pydantic event models before this handler is merged. If they are not present, create them as part of this SI — do not invent a new schema file.

**Idempotency:** Stripe may deliver the same webhook event more than once. Consider checking whether the instalment schedule is already in a terminal state before processing — a second `succeeded` event for a paid instalment should be a no-op, not a double-emit.

**Endpoint registration:** After deployment to each environment, the endpoint URL must be registered in the Stripe dashboard. This is an operational step — add it to the deployment runbook, not the code.

**Tests:** Test signature verification failure (returns 400), `succeeded` path (status updated, event emitted), `failed` path (status updated, event emitted), unrecognised event type (returns 200, no action).
