# Expose instalment checkout endpoint

## Service

**order-service** — REST service managing the full order lifecycle — basket, checkout, payment and fulfilment handoff. Python 3.12 / FastAPI / asyncpg / PostgreSQL 15 / Stripe SDK 9.x.

## Phase

Phase 3 — Checkout endpoint

## Description

Add `POST /orders/{id}/checkout/instalment` to `orders/router.py`. The handler must: validate the request (reject if the order is already paid or cancelled), call `InstalmentCalculator` (SI-03) with the order total and requested term, call `InstalmentGatewayClient.create_instalment_intent` (SI-02) with the computed schedule, persist the resulting `InstalmentPlan` and `InstalmentSchedule` records (SI-01 repository), and return HTTP 201 with the schedule and gateway payment reference. Depends on SI-02 and SI-03 both being merged.

- Blocks: SI-05

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Location:** `src/orders/router.py` — add a new route registration for `POST /orders/{id}/checkout/instalment`.

**Validation guards:** Before calling the calculator or gateway, query the order state via the existing repository and raise `HTTP 422` (or `HTTP 409`) if the order status is `paid` or `cancelled`. The IA is explicit that these must be rejected.

**Transactionality:** The gateway call and the DB persist should be treated as a two-phase commit problem — if the DB write fails after the gateway intent is created, you will have a dangling Stripe intent. Consider whether the gateway intent should be created first (and cancelled on DB failure) or whether an idempotency key can handle retries safely. Document whichever approach is chosen.

**Response schema:** Return a Pydantic response model (consistent with the existing router pattern) — do not return raw dicts. Include `instalment_plan_id`, `schedule` (list of amounts/dates), and `gateway_reference`.

**Tests:** Test happy path (201 response), already-paid rejection, already-cancelled rejection, and calculator/gateway error propagation.
