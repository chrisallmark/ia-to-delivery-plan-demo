# Add delivery status webhook

## Service

**notification-service** — REST service that dispatches transactional notifications across email, SMS, and (soon) push channels. Node.js 20 / TypeScript 5 / Fastify 4 / PostgreSQL 15.

## Phase

Phase 4 — Delivery webhook

## Description

Add `POST /notifications/push/status` to receive FCM delivery receipts. The handler must: validate the incoming FCM webhook payload; update the delivery record for the notification (persisted by SI-04); and emit a domain event via `src/events/emit.ts`. The event schema must match the existing email/SMS domain events defined in `src/events/schema.ts` — do not invent a new schema. Depends on SI-04 having written the delivery record that this handler updates.

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Location:** `src/routes/` — add the new route to the push routes file created in SI-04 (`src/routes/push.ts`) or as a separate file, consistent with how email/SMS status webhooks are structured (check `src/routes/` for existing patterns).

**Event schema:** Review `src/events/schema.ts` before writing the emit call. The IA is explicit that the push delivery event schema must match existing email/SMS event schemas. If the schema needs a new event type (e.g. `notification.push.delivered`, `notification.push.failed`), add it to `src/events/schema.ts` rather than emitting an ad-hoc object.

**FCM receipt format:** FCM delivery receipts have a specific payload format. Read the Firebase documentation for the webhook receipt structure before writing the payload parser — do not assume it matches the Stripe webhook pattern from the order-service.

**Idempotency:** FCM may deliver the same receipt more than once. Check whether the delivery record is already in a terminal state before updating and emitting. A second receipt for an already-delivered notification should be a no-op.

**Tests:** Test successful delivery receipt (record updated, event emitted), failed delivery receipt (record updated, event emitted), duplicate receipt (no-op), and invalid payload (400).
