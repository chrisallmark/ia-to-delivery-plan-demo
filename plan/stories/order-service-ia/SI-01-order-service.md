# Add instalment plan data model

## Service

**order-service** — REST service managing the full order lifecycle (basket, checkout, payment, fulfilment handoff). Owned by the Payments team.

## Phase

Phase 1 — Foundation

## Description

Introduce `instalment_plans` and `instalment_schedules` tables to the PostgreSQL database via a numbered migration file. Add the corresponding dataclass domain models to `src/orders/models.py` and async repository query helpers to `src/orders/repository.py`. This is the foundational layer for all instalment work; no other SI may begin until these tables, models, and repository methods are present and merged.

**Migration:** Add a new numbered `.sql` file to `db/migrations/` (e.g. `003_add_instalment_tables.sql`) and register it in `db/migrations/__init__.py`. The migration creates `instalment_plans` (plan-level data: order reference, term, APR, gateway reference) and `instalment_schedules` (individual payment rows: due date, amount, status, paid-at timestamp). Production migrations are applied manually via the `migrate` CLI.

**Domain models:** Add `InstalmentPlan` and `InstalmentSchedule` dataclasses to `src/orders/models.py` following the existing patterns (no ORM — pure Python dataclasses).

**Repository:** Add `InstalmentRepository` to `src/orders/repository.py` with at minimum `create_plan`, `create_schedule`, `get_plan_by_order`, and `get_schedules_by_plan` methods. All DB access uses raw asyncpg SQL — no query builder.

**Conventions to follow:** Repository methods must be async. Use the existing asyncpg pool from `db/connection.py`. Do not use SQLAlchemy or any ORM construct.
