# Handle instalment payment webhooks

## Service

**order-service** — REST service managing the full order lifecycle (basket, checkout, payment, fulfilment handoff). Owned by the Payments team.

## Phase

Phase 4 — Webhooks

## Description

Add `POST /webhooks/stripe/instalment` to `src/payments/router.py`. The handler must: verify the `Stripe-Signature` header using `stripe.Webhook.construct_event()` (return `400` on failure); handle two event types — `payment_intent.succeeded` (update the matching `InstallmentSchedule` to `paid`, set `paid_at`) and `payment_intent.payment_failed` (update status to `failed`); emit a domain event per instalment via `events.emit.publish()` using event types `instalment.paid` and `instalment.failed` added to `events/schema.py`. Requires `STRIPE_WEBHOOK_SECRET` from environment variables. The endpoint must be registered in the Stripe dashboard per environment before it receives events.

## Implementation hint

**Target file:** `src/payments/router.py` — add alongside existing Stripe webhook routes.

**Signature verification:** Must call `stripe.Webhook.construct_event(payload, sig_header, STRIPE_WEBHOOK_SECRET)` before any processing. A missing or invalid signature must return `400` immediately — do not process the body. `STRIPE_WEBHOOK_SECRET` is a per-environment value injected via environment variable.

**Event type dispatch:** After verification, inspect `event.type`. For `payment_intent.succeeded`: look up the `InstallmentSchedule` by `gateway_payment_intent_id`, set `status = 'paid'` and `paid_at = now()`, persist via `InstalmentRepository`. For `payment_intent.payment_failed`: set `status = 'failed'`, persist. Return `200` for both handled event types. Return `200` (with no-op body) for unrecognised event types — do not return `4xx` for unknown types or Stripe will retry indefinitely.

**Schema extension:** Before implementing the handler, extend `events/schema.py` with `InstalmentPaidEvent` and `InstalmentFailedEvent` versioned Pydantic models. Emit via `events.emit.publish()` after the DB write succeeds.

**Tests:** Cover: valid signature + succeeded event, valid signature + failed event, invalid signature (expect 400), unknown event type (expect 200 no-op).

**Environment setup:** Document in the service README that `POST /webhooks/stripe/instalment` must be registered in the Stripe dashboard for dev, staging, and prod. Failure to do this results in silent event loss.
