# Expose instalment checkout endpoint

## Service

**order-service** — REST service managing the full order lifecycle (basket, checkout, payment, fulfilment handoff). Owned by the Payments team.

## Phase

Phase 3 — Checkout Endpoint

## Description

Add `POST /orders/{id}/checkout/instalment` to `src/orders/router.py`. The endpoint must: validate the request body (term length, APR); reject if the order is already paid or cancelled (return `409`); call `InstalmentCalculator.calculate()` to produce the schedule; call `InstalmentGatewayClient.create_instalment_intent()` to obtain a gateway reference; persist the plan and schedule via `InstalmentRepository`; return `201` with the schedule and gateway reference. Depends on SI-02 (gateway) and SI-03 (calculator) being merged.

**Target file:** `src/orders/router.py` — add the new route alongside existing checkout routes.

**Request validation:** Use a Pydantic model for the request body (term_months: int, apr: Decimal). Validate `term_months` is positive and `apr` is within a plausible range.

**Order state guard:** Fetch the order via the existing `OrderRepository` before proceeding. Return `409 Conflict` if `order.status` is `paid` or `cancelled`.

**Orchestration:** Call `InstalmentCalculator.calculate()`, then `InstalmentGatewayClient.create_instalment_intent()`, then `InstalmentRepository.create_plan()` and `create_schedule()` in a single transaction. If the gateway call fails, do not persist — return `502`.

**Response:** Return HTTP `201` with a JSON body containing the serialised `InstalmentPlan` (including gateway reference) and the list of `InstalmentSchedule` items.

**Tests:** Cover the happy path, the already-paid rejection, the cancelled rejection, and a gateway failure.
