# Add push provider abstraction

## Service

**notification-service** — REST service that dispatches transactional notifications across email, SMS, and (soon) push channels. Node.js 20 / TypeScript 5 / Fastify 4 / PostgreSQL 15.

## Phase

Phase 1 — Foundation

## Description

Introduce a `PushProvider` interface and a factory that resolves the correct push provider at runtime. The current codebase hard-codes email and SMS channels; this SI adds the push abstraction without any concrete implementation. The interface must follow the existing `BaseChannel` pattern so the channel factory in `src/channels/index.ts` can resolve it by name. All subsequent push SIs (SI-02 through SI-05) depend on this interface being merged. No new routes, database changes, or external SDK calls in this SI.

## Dependencies

- Blocked by: none
- Blocks: SI-02, SI-03, SI-04, SI-05

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md. It is a starting point, not a specification. Verify against the
> current codebase before starting work.

**Location:** `src/channels/push/` — create a new subfolder following the existing `src/channels/email/` and `src/channels/sms/` pattern. Add an `index.ts` that exports a class implementing `src/channels/BaseChannel.ts`.

**Factory registration:** Register the new push channel in `src/channels/index.ts`. This is the integration point — if the factory does not know about the push channel, SI-04's endpoint will fail at runtime even if the provider is implemented.

**Interface only at this stage:** The concrete `FirebasePushProvider` (SI-02) is intentionally deferred. This SI should export either an abstract base class or a no-op stub that satisfies the interface so the factory compiles and tests pass without Firebase credentials.

**Tests:** Verify the factory resolves `"push"` to the stub provider. TypeScript compilation (`pnpm build`) must pass — treat type errors as test failures.
