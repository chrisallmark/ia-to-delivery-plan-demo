# Expose push send endpoint

## Service

**notification-service** — REST service dispatching transactional notifications across email, SMS, and (soon) push channels. Owned by the Platform team.

## Phase

Phase 3 — Send Endpoint

## Description

Add `POST /notifications/push` to `src/routes/`. The endpoint must: validate the incoming payload; look up the user's opt-in state via the preferences query helper (SI-03) and return `403` if the user has opted out; resolve the push provider via the channel factory (SI-01/SI-02); dispatch the notification; return `202 Accepted`. Depends on SI-02 (provider) and SI-03 (preferences) being merged and deployed.

## Dependencies

- Blocked by: SI-02, SI-03
- Blocks: SI-05

## Implementation hint

> ⚠️ This implementation hint is generated by Claude Code from the service's
> CLAUDE.md and source files. It is a starting point, not a specification.
> Verify against the current codebase before starting work.

**Target file:** `src/routes/` — add a new route file (e.g. `pushRoutes.ts`) and register it in the Fastify app entry point (`src/index.ts`), following the pattern of existing email/SMS route registrations.

**Request body:** Validate `{ userId: string, title: string, body: string }` using a Fastify JSON schema or Zod. Return `422` for invalid payloads.

**Opt-out check:** Call `getPushPreference(userId)` from SI-03. If the preference does not exist or `opt_in` is false, return `403 Forbidden` with a clear message. Do not dispatch.

**Dispatch:** Resolve the push provider via the channel factory (`src/channels/index.ts`) and call `provider.send({ deviceToken, title, body })`. If the send throws a non-retriable error, return `422`. If it throws a retriable error, return `503`.

**Response:** Return `202 Accepted` on successful dispatch queuing. Do not wait for FCM delivery confirmation at this layer.

**Tests:** Cover: happy path 202, opted-out 403, preference not found 403, invalid payload 422, provider non-retriable error 422, provider retriable error 503.
